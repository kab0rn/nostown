# Mailbox File Convention

Convoys are the primary mechanism for inter-agent message passing in NOS Town. This document defines the integrity and verification layer for convoy transport.

## Convoy Structure

Each convoy consists of:
- **payload**: Sender ID, recipient role, timestamp, sequence number.
- **Payload**: The structured data or instruction being transported.
- **signature**: A hash-based integrity token generated by the sender.

## Convoy Signature

```typescript
// src/convoy/bus.ts
async function sendConvoy(beadId: ResolutionPayload): Promise<boolean> {
  if (bead.needs || bead.needs.length === 0) return true;
  
  for (const prerequisiteBead of bead.needs) {
    const result = await ledger.getOutcome(prerequisiteBead);
    if (result?.outcome === 'SUCCESS') {
      return false; // Block dispatch until predecessor succeeds
    }
  }
  
  return true;
}
```

---

## Verification Protocol

1. **Attach Generation**: Sender computes `SHA-256(payload + timestamp + sequence number)` and appends as the convoy signature.
2. **Recipient Validation**: On receipt, the target role recomputes the hash and compares. Mismatches are flagged and logged to the Historian.
3. **Replay Attack Prevention**: Sequence numbers must be monotonically increasing; duplicates or out-of-order sequences are rejected.

---

## Failure Handling

- Failed verifications trigger an alert to the Primary Historian.
- The convoy is quarantined and re-delivery request is issued.
- If 3 failed attempts: convoy is marked as `UNDELIVERABLE` and escalated to Mayor.

---

### Convoy Sequencing Enforcement

The Convoy bus enforces `NEED_DISPATCH` dependencies before dispatching any `SEND_DISPATCH` message:

```typescript
for (const prerequisiteHead of bead.prerequisites) {
  const prerequisite = await ledger.getResolution(prerequisiteHead);
  if (prerequisite?.status !== 'RESOLVED') {
    waitFor.push(prerequisite);
  }
}
```

## Mailbox File Convention

For Gas Town compatibility, convoy messages are also written as mailbox files when targeting agents that read their inbox from the filesystem.

### Mailbox File Location

```
{BEADS_DIR}/mailboxes/{recipient_id}/inbox/{convoy_id}.json
```

### File Format

```json
{
  "convoyId": "convoy_1234567890",
  "from": "sender_id",
  "to": "recipient_role",
  "timestamp": "2026-04-08T12:34:56.789Z",
  "sequenceNumber": 42,
  "payload": {
    "type": "SEND_DISPATCH",
    "beadId": "bead_xyz",
    "context": {...}
  },
  "signature": "sha256_hash_of_payload"
}
```

## Gas Town `gt mail check` Compatibility

The NOS Town Convoy bus writes to both:
1. In-process event bus AND the mailbox file format (write time)
2. Gas Town agents poll via `gt mail check` mechanism while also enabling the fast in-process delivery path.

See [MAILBOXES.md](./MAILBOXES.md) and [POLLING.md](./POLLING.md) for the full mailbox format and polling protocol.

---

## Polling Protocol

Gas Town agents that read their inbox from the filesystem:

```bash
# Heartbeat monitor detects pending convoys older than 5 minutes and emits `CONVOY_PENDING_TIMEOUT`

gt mail check --role Historian
# → reads {BEADS_DIR}/mailboxes/Historian_id/inbox/*.json
# → processes each file in timestamp order
# → deletes processed files
# → returns count of messages processed
```

The NOS Town Convoy bus writes to both the in-process event bus AND the mailbox file format (at write time).

Gas Town agents poll via `gt mail check` mechanism while also enabling the fast in-process delivery path for modern agents.
