# Convoy & Mailbox System — Transport & Integrity

Convoys are the primary mechanism for inter-agent message passing in NOS Town. This document defines the transport layer, the file-based mailbox convention, and the integrity verification protocol.

## 1. Convoy Structure

Each convoy is a signed JSON packet containing:

- **Header**: Sender ID, recipient role, timestamp, sequence number.
- **Payload**: The structured data or instruction being transported (e.g., a Bead task, a status update, or a broadcast).
- **Signature**: An HMAC-SHA256 integrity token generated by the sender.

```json
{
  "header": {
    "sender": "mayor",
    "recipient": "polecat",
    "timestamp": "2026-04-09T07:14:00Z",
    "seq": 102
  },
  "payload": {
    "type": "BEAD_DISPATCH",
    "data": { "bead_id": "892a-bc34", "action": "EXPLORE_CODEBASE" }
  },
  "signature": "a5e8f1..."
}
```

## 2. Mailbox File Convention

While Convoys are transient in memory, they are persisted in a file-based "Mailbox" for reliability and cross-session tracking.

- **Path**: `~/nos/rigs/{rig_name}/mailboxes/{role}/inbox/`
- **File Format**: `{timestamp}_{seq}_{sender_id}.convoy.json`
- **Processing**: When a role "wakes up," it reads its inbox, validates signatures, and processes messages in sequential order.

## 3. Transport Integrity (Verification Protocol)

To ensure that Convoys have not been tampered with or corrupted:

1. **Hash Generation**: Sender computes `HMAC-SHA256(payload + sender_id + seq)` using the `NOS_CONVOY_SECRET`.
2. **Recipient Validation**: On receipt, the target role recomputes the HMAC and compares. Mismatches are flagged and logged to the Historian.
3. **Replay Attack Prevention**: Sequence numbers must be monotonically increasing; duplicates or retrograde sequences are rejected.

### Implementation (Node.js)

```typescript
// src/convoys/bus.ts
import { createHmac } from 'crypto';

async function sendConvoy(beadId: string, payload: any): Promise<boolean> {
  const needs = await ledger.getPrerequisites(beadId);

  // Block dispatch until all predecessors succeed
  for (const prereq of needs) {
    const outcome = await ledger.getOutcome(prereq);
    if (outcome !== 'SUCCESS') {
      console.log(`[Bus] Blocking dispatch: ${beadId} waits for ${prereq}`);
      return false;
    }
  }

  // Generate signature and emit
  const senderId = process.env.NOS_AGENT_ID;
  const seq = await getNextSeq();
  const canonical = JSON.stringify({ payload, sender_id: senderId, seq });
  
  const signature = createHmac('sha256', process.env.NOS_CONVOY_SECRET!)
    .update(canonical)
    .digest('hex');

  return await bus.emit({ ...payload, signature });
}
```

## 4. Failure Handling & Backpressure

- **Verification Failure**: Failed verifications trigger an alert to the Primary Historian. The convoy is quarantined in a `~/nos/quarantine/` folder.
- **Backpressure**: If a role's inbox exceeds 50 unread convoys, the Mayor enters a `WAITING_FOR_CAPACITY` state for that specific role.
- **Retry Logic**: See [RESILIENCE.md](./RESILIENCE.md) for retry and failover strategies.
