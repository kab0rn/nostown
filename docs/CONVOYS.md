# Convoy & Mailbox System — Transport & Integrity

Convoys are the primary mechanism for inter-agent message passing in NOS Town. This document defines the transport layer, the file-based mailbox convention, and the integrity verification protocol.

## 1. Convoy Structure

Each convoy is a signed JSON packet containing:

- **Header**: Sender ID, recipient role, timestamp, sequence number.
- **Payload**: The structured data or instruction being transported (e.g., a Bead task, a status update, or a broadcast).
- **Signature**: An Ed25519 signature generated by the sender role instance.
- **Transport MAC (optional)**: An HMAC-SHA256 token generated by the transport layer for fast cluster-local tamper detection.

```json
{
  "header": {
    "sender_id": "mayor_01",
    "recipient": "polecat",
    "timestamp": "2026-04-09T07:14:00Z",
    "seq": 102,
    "trace_id": "8e42...f2a",
    "parent_span_id": "0f2a...12b"
  },
  "payload": {
    "type": "BEAD_DISPATCH",
    "data": {
      "bead_id": "892a-bc34",
      "action": "EXPLORE_CODEBASE",
      "plan_checkpoint_id": "plan_7ab2",
      "critical_path": true,
      "fan_out_weight": 6,
      "priority": "high"
    }
  },
  "signature": "ed25519:base64...",
  "transport_mac": "hmac256:hex..."
}
```

## 2. Mailbox File Convention

While Convoys are transient in memory, they are persisted in a file-based "Mailbox" for reliability and cross-session tracking.

- **Path**: `~/nos/rigs/{rig_name}/mailboxes/{role}/inbox/`
- **File Format**: `{timestamp}_{seq}_{sender_id}.convoy.json`
- **Processing**: When a role "wakes up," it reads its inbox, validates signatures, and processes messages in sequential order.

## 3. Transport Integrity & Identity

NOS Town uses a two-layer trust model:

1. **Sender authenticity** — every role instance signs the convoy canonical form with its own Ed25519 private key.
2. **Transport integrity** — the local cluster transport MAY attach an HMAC using `NOS_CONVOY_SECRET` for fast tamper detection in trusted deployments.

Shared HMAC alone is NOT sufficient to establish sender identity. A compromised agent with access to the shared secret could otherwise forge a Mayor or Safeguard message.

To ensure that Convoys have not been tampered with or corrupted:

1. **Canonicalization**: Sender canonicalizes `{header, payload}` before signing.
2. **Sender Signature**: Sender computes `Ed25519.sign(canonical_payload)` using its per-role private key.
3. **Optional Transport MAC**: The transport layer computes `HMAC-SHA256(canonical_payload)` using `NOS_CONVOY_SECRET`.
4. **Recipient Validation**:
   - verify sender public key against `sender_id`
   - verify optional transport MAC if present
   - verify `seq` monotonicity
   - verify sender is authorized for `payload.type`
5. **Replay Attack Prevention**: Sequence numbers must be monotonically increasing per sender; duplicates or retrograde sequences are rejected.

### 3.1 Sender Authorization Matrix

| Sender Role | Allowed Types |
|---|---|
| Mayor | `BEAD_DISPATCH`, `SWARM_ABORT`, `CAPACITY_UPDATE`, `LOCKDOWN_BROADCAST` |
| Polecat | `BEAD_STATUS`, `DISCOVERY`, `BLOCKED`, `PATCH_READY` |
| Witness | `REVIEW_VERDICT`, `COUNCIL_VOTE`, `REVIEW_RETRY` |
| Safeguard | `SECURITY_VIOLATION`, `LOCKDOWN_TRIGGERED`, `WRITE_APPROVED`, `WRITE_REJECTED` |
| Historian | `ROUTING_UPDATE`, `PLAYBOOK_PUBLISHED`, `BACKFILL_NOTICE` |

Any validly-signed convoy with an unauthorized payload type is rejected with `AUTHZ_DENIED`.

### Implementation (Node.js)

```typescript
// src/convoys/bus.ts
import { createHmac } from 'crypto';

async function sendConvoy(beadId: string, payload: any): Promise<boolean> {
  const needs = await ledger.getPrerequisites(beadId);

  // Block dispatch until all predecessors succeed
  for (const prereq of needs) {
    const outcome = await ledger.getOutcome(prereq);
    if (outcome !== 'SUCCESS') {
      console.log(`[Bus] Blocking dispatch: ${beadId} waits for ${prereq}`);
      return false;
    }
  }

  // Generate signature and emit
  const senderId = process.env.NOS_AGENT_ID;
  const seq = await getNextSeq();
  const canonical = canonicalize({ payload, sender_id: senderId, seq });
  const signature = signWithRoleKey(canonical, senderId);
  const transportMac = process.env.NOS_CONVOY_SECRET
    ? createHmac('sha256', process.env.NOS_CONVOY_SECRET).update(canonical).digest('hex')
    : undefined;

  return await bus.emit({ ...payload, signature, transport_mac: transportMac });
}
```

## 4. Failure Handling & Backpressure

- **Verification Failure**: Failed verifications trigger an alert to the Primary Historian. The convoy is quarantined in a `~/nos/quarantine/` folder.
- **Backpressure**: If a role's inbox exceeds 50 unread convoys, the Mayor enters a `WAITING_FOR_CAPACITY` state for that specific role.
- **Priority-aware draining**: Queue draining is not FIFO-only. Convoys are drained in this order:
  1. `critical_path = true`
  2. highest `fan_out_weight`
  3. explicit `priority`
  4. FIFO within same class
- **Checkpoint gating**: `BEAD_DISPATCH` convoys MUST include a valid `plan_checkpoint_id`. Missing or unresolved checkpoint IDs are rejected with `MAYOR_CHECKPOINT_MISSING`.
- **Retry Logic**: See [RESILIENCE.md](./RESILIENCE.md) for retry and failover strategies.
